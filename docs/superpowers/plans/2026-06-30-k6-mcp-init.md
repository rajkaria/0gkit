# K6 ‚Äî `0g mcp init <agent>`

## Reality check (2026-07-01) ‚Äî what the original draft got wrong

The first draft was reconciled against the real code before coding (the K1/K5
discipline). Four drifts, one structural:

1. **Decision numbers.** Draft claimed the new decisions were **D84‚ÄìD85**; K5
   already shipped **D84/D85/D86** (`docs/DECISIONS.md`). ‚Üí Renumbered to
   **D87‚ÄìD88**.
2. **`OGKIT_MCP_KITS` env (structural).** Draft would write this env "so the
   running server enables those tools". Reality: the neutral server
   (`packages/0gkit-mcp/src/server.ts`) serves a **fixed** `[...TOOLS]` (9 `og_*`
   tools) + optional foundry plugin ‚Äî **nothing reads `OGKIT_MCP_KITS`**. ‚Üí Env
   var **eliminated**; kit tools ride the real plugin seam instead (below).
3. **`adapters["mcp-agent"]` misread (structural).** Draft treated a non-empty
   `tiers.adapters["mcp-agent"]` as "kit ships MCP tools". Reality: `mcp-agent`
   is a **base template** (peer of `react-app`, `chat`); that adapter is files
   copied when the kit is applied **onto the mcp-agent scaffold**, wiring into
   the **user's own project server**, not the published
   `npx @foundryprotocol/0gkit-mcp` binary. ‚Üí Synergy is delivered **inside the
   user's kitted project**, not by mutating the published server.
4. **Adapter API mismatch (structural).** Draft assumed
   `registerXTools(server, ‚Ä¶)` wires into `create0gMcpServer()`. Reality:
   adapters target the **high-level `McpServer.tool()`** API (`McpServerLike`),
   but `create0gMcpServer()` returns the **low-level `Server`** (no `.tool()`) ‚Äî
   the shipped adapters were **never actually wireable** (a latent K1‚ÄìK4 gap). ‚Üí
   K6 reconciles it via a neutral `collectToolPlugin()` adapter over the existing
   `{ name, tools, call }` plugin seam.

**Net:** the neutral config-writer half of K6 is sound and valuable. The
"synergy" half is redesigned around the real plugin seam and delivered honestly.

## Goal

Make 0gkit **native to your AI tool** in one command:

```bash
0g mcp init cursor        # writes .cursor/mcp.json
0g mcp init claude        # writes ./.mcp.json (project)
0g mcp init windsurf      # writes .windsurf/mcp.json
0g mcp init codex         # writes .codex/mcp.json
0g mcp init claude --global   # writes the agent's user-level config instead
```

By default the config points the editor at the published neutral server
(`npx -y @foundryprotocol/0gkit-mcp`, the 9 `og_*` tools).

**Honest K0 synergy:** when run inside a project scaffolded from the
**`mcp-agent` base** that has kits applied (`.0gkit/kits.json.base === "mcp-agent"`
and applied kits ship an `mcp-agent` adapter), `0g mcp init` instead points the
editor at **that local project's server**, which ‚Äî via a generated `src/kits.ts`
aggregator ‚Äî exposes the kit tools (`memory_remember`, `oracle_ask`, ‚Ä¶) **on top
of** the 9 neutral tools. The kit tools appear because the editor runs the
user's own kitted server, not because an env var is faked into the neutral one.

## Architecture / real seams (all verified against code)

- **Neutral plugin seam already exists.** `create0gMcpServer()` merges an
  optional `foundryPlugin: { name, tools: Tool[], call(name,args) }` into its
  tool list and routes calls to it (`server.ts:44,53`). K6 **generalizes** this
  to `plugins?: McpToolPlugin[]` (foundry becomes one plugin internally;
  `foundryPlugin` option kept for back-compat).
- **`collectToolPlugin(name, register, opts?)`** ‚Äî new **neutral, generic**
  helper in `0gkit-mcp`. It hands the kit adapter's existing
  `register*Tools(server: McpServerLike, opts)` a tiny collector implementing
  `McpServerLike.tool(name, description, schema, handler)`; it captures the
  tools + handlers and returns an `McpToolPlugin` `{ name, tools, call }`. This
  adapts the shipped adapters **as-is** (no rewrite of their bodies) and imports
  **no kit overlay** ‚Äî neutrality (D78) preserved.
- **Kit adapters gain one additive export.** Each of the 4 mcp-agent adapters
  (`agent-memory`, `ai-oracle`, `durable-agent`, `sealed-inference`) adds
  `export const mcpToolPlugin = (env) => collectToolPlugin("<kit>", (s) => registerXTools(s, {‚Ä¶env}))`
  next to (not replacing) its `registerXTools`. Fixed export name ‚áí generatable.
- **Aggregator generated by `applyKit`.** When `base === "mcp-agent"`, after
  copying adapter files `applyKit` (re)writes `src/kits.ts`: imports
  `mcpToolPlugin as <kit>Plugin` from each applied kit's declared adapter path
  (`kit.json.tiers.adapters["mcp-agent"][0]`), exports `kitPlugins = [...]`.
  Union/idempotent across re-applies. The template ships a **stub** `src/kits.ts`
  (`export const kitPlugins = []`) so a bare mcp-agent project builds.
- **Template consumes it.** `templates/mcp-agent/src/index.ts` ‚Üí
  `create0gMcpServer({ plugins: kitPlugins })`; stderr tool list mentions kit
  tools when present.
- **`0g mcp init`** is a thin CLI shell (lazy-imports `0gkit-mcp`, D39). It reads
  `.0gkit/kits.json` via the injected `ProgramDeps.fs`; picks **neutral** (npx)
  or **local** (`npm start` in the project) server command; writes via
  `deps.fs`. `--global` ‚Üí user-level path.
- **`.0gkit/kits.json` shape is `{ applied: string[], base: string, at: string }`**
  (verified in `apply.ts:312`) ‚Äî `base` drives local-vs-npx detection.
- New docs page `apps/docs/app/concepts/mcp/page.mdx`.

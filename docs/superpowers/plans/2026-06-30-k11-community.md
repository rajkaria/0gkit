# K11 — Community surface

## Goal

Stand up the community surface: enable GitHub Discussions on `rajkaria/0gkit`,
seed + pin the categories, and make the landing footer Community links real.
After K11:

- Discussions enabled with **Q&A / Show and tell / Ideas / RFCs / Show your kit**.
- Each category seeded with a welcome post; a pinned "How to ask great questions"
  post points at the `--copy-issue-context` flow (SP15).
- Landing footer Community column links Discussions + Telegram (Discord deferred).

**K4 synergy:** the **"Show your kit"** category seeds community kits and ties to
K4's `docs/kits/AUTHORING.md` — the engine being general makes community kits a
real, ownable GTM surface.

## Dependencies / Architecture

- **Mostly config + a small landing PR** (the roadmap's lowest-effort sprint).
- **GitHub Discussions** is enabled via repo settings / the `gh` API. Categories
  and pinned posts are created via `gh api`. No package code.
- **The landing footer already has a Community column**
  ([apps/landing/components/Footer.tsx](../../../apps/landing/components/Footer.tsx))
  linking GitHub / Discussions / Issues / Contributing / Telegram / Changelog.
  K11 (a) verifies those links resolve once Discussions is live, and (b) adds a
  direct **"Show your kit"** category link so the authoring-guide loop is one
  click from the landing page.
- **Pinned "how to ask"** post references the shipped `--copy-issue-context` CLI
  flag (SP15) so issues/discussions arrive with a clean, redacted context block.
- **No paid tiers** (out-of-scope invariant); Discord is **deferred** until
  Discussions traction warrants it.

## Tech Stack

GitHub Discussions (config via `gh api`), Markdown seed posts, Next.js landing
(`apps/landing`). No published-package change.

### T1 — enable Discussions + create the categories

- [ ] **Implement** — `scripts/setup-discussions.sh`: enable Discussions on
      `rajkaria/0gkit` and create the five categories — **Q&A** (answerable), **Show
      and tell**, **Ideas**, **RFCs**, **Show your kit** (answerable=false; the K4
      synergy). Use `gh api` GraphQL; the script is **idempotent** (skips categories
      that already exist). Example shape:

```bash
#!/usr/bin/env bash
set -euo pipefail
REPO="rajkaria/0gkit"
# 1. ensure Discussions is enabled
gh api -X PATCH "repos/$REPO" -f has_discussions=true >/dev/null
# 2. create categories that don't exist yet (GraphQL — see gh api graphql)
for cat in "Q&A:true" "Show and tell:false" "Ideas:false" "RFCs:false" "Show your kit:false"; do
  name="${cat%%:*}"; answerable="${cat##*:}"
  # … createDiscussionCategory mutation, guarded by a list-existing check …
done
```

(Document the exact GraphQL mutation in the script comments; the script is the
runbook, not a one-shot.)
